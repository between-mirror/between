Security

The useful version of this page is the part that says what is not protected. A security page that only lists strengths is marketing.

Reporting a vulnerability

Please do not open a public issue. Use a GitHub security advisory, which stays private until a fix ships. Acknowledged within 48 hours, fixed and released as a patch, then published with an honest affected-version range.

What is actually enforced

Each of these is an enforced invariant with a regression test — that is the difference between a promise and a control the build continuously checks:

  • Loopback-only. The server refuses to boot on a non-loopback host, and rejects a rebinding Host or a foreign Origin.
  • No telemetry. Zero analytics dependencies, checked in the build.
  • Server-side engine enforcement. The client cannot choose an engine outside your mode; there is no silent fallback to the test mock, which exists only under a test flag.
  • The evidence chain. Model results are verified against the job they claim to answer, re-validated after cleaning, and every claim's receipts must resolve to real messages or the claim is dropped.
  • Immutable releases. A version identifies one tree forever. Corrections are a new patch version; tags never move.

What is not protected

Anyone with your unlocked computer has your archive. There is no app password, and adding one would be theatre against someone already at your desk.

No encryption at rest. Deliberately deferred: on a tool with no accounts and no recovery, a forgotten passphrase means losing something irreplaceable. Best-effort owner-only file permissions and a loud warning if the folder looks cloud-synced are the current posture.

A cloud-synced folder defeats all of this. If the app sits in OneDrive or Dropbox, your archive is on someone else's servers regardless of what this program does.

Model containment is tool-level, not OS-level. When the reading runs through a Claude subscription it runs with restricted tools in a staged temp folder holding only the pending jobs — no database, no archive, no repository, no home directory. That is containment by configuration. A real operating-system boundary arrives with the packaged desktop app, and until then this is described exactly as it is.

Archive text is untrusted input. Someone who sent you messages years ago could have written something intended to manipulate a model reading them later. Under the documented Claude Code tool-policy assumptions, the intended prompt-injection ceiling is a dishonest result file. Because this is not OS-level isolation, Between Mirror does not claim protection against a failure in that containment layer.

A review found real defects, and here is what happened

An adversarial security and grounding review in July 2026 found that several core promises were made by convention rather than by mechanism, and that a few were quietly false. Promotion was stopped and the boundaries were rebuilt first. Every original finding is fixed with a regression test; what remains open is listed as open.

The current status — fixed, deferred by design, and newly tracked — is kept in STATUS.md and updated with every release. A review is a snapshot, not a certificate.

Provenance

Every release is an immutable tag and a commit SHA on the public repository, published through a single scripted path that refuses to publish if a version would name a second tree. Signed installers, checksums, an SBOM and build attestation arrive with the packaged desktop app.
